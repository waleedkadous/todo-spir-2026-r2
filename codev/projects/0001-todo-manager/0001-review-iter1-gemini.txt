I found two minor edge cases that don't block release but are worth noting for future refinement:

1.  **Empty State Messaging**: The `TodoList` component checks `status` and `priority` filters but overlooks `dueBefore`/`dueAfter` when determining if filters are active. This means if a user filters by date and sees no results, the empty state says "No todos yet" (as if the list is empty) rather than "No matching todos".
2.  **Error Handling Granularity**: In `useChat.ts`, the `try/catch` block wraps both the network request and the local action execution. If `executeAction` throws an error (e.g., due to a race condition where a todo was deleted manually while the AI was thinking), the catch block defaults to "Failed to connect to the server", which is misleading.

These are minor polish issues. The core functionality is solid.
